# Project Memory

在开始实现前，先快速阅读 `docs/ai-memory/README.md`，按需再读其中引用的专题文件。

在完成任何非微小改动后，更新项目记忆文件。不要只改代码不留记录。

## 读取顺序

按最小必要原则读取，避免一次性加载所有记忆文件：

1. 先读 `docs/ai-memory/README.md`
2. 如果任务涉及长期约定、用户偏好或容易遗漏的细节，再读 `docs/ai-memory/product-context.md`
3. 如果任务涉及已有实现模式、模块边界或历史坑点，再读 `docs/ai-memory/engineering-context.md`
4. 如果任务需要了解最近几次 AI 改动，再读 `docs/ai-memory/change-log.md`
5. 如果任务有明确未完事项或用户要求后续跟进，再读 `docs/ai-memory/open-questions.md`

## 更新触发条件

出现以下任一情况时，必须更新至少一个记忆文件：

1. 修改了业务逻辑、接口、数据结构、配置、脚本或构建流程
2. 用户提出了值得长期保留的细节要求、偏好或禁忌
3. 发现了会反复踩坑的限制、隐式约定或实现细节
4. 做出了带取舍的实现决策，后续需要知道为什么这样做
5. 留下了明确的后续事项、风险、待确认项

仅做纯格式化、无行为变化的注释修正、或临时探索且未落地时，可以不更新；但如果用户在过程中补充了长期有效的要求，仍然要记录。

## 文件分工

只更新必要文件，避免重复写相同信息：

### `docs/ai-memory/README.md`

维护短入口：

1. 当前项目是什么
2. 后续 AI 开工前必须先知道什么
3. 该读哪几个文件

只有当入口本身失真时才更新。

### `docs/ai-memory/product-context.md`

记录稳定且高价值的产品/需求信息：

1. 用户长期偏好
2. 业务规则
3. 命名或交互约定
4. 明确不希望发生的事情

不要写一次性实施细节。

### `docs/ai-memory/engineering-context.md`

记录稳定的工程实现知识：

1. 目录职责与模块边界
2. 推荐扩展点
3. 容易误改的实现约束
4. 已验证可行的模式
5. 已知坑点与规避方式

不要把每次小改动都堆到这里；这里只保留未来高概率复用的信息。

### `docs/ai-memory/change-log.md`

按时间倒序追加重要改动：

1. 日期
2. 改了什么
3. 为什么改
4. 影响范围
5. 是否有风险或后续动作

每次条目尽量控制在几行内，可扫描即可。

### `docs/ai-memory/open-questions.md`

记录尚未完成闭环的信息：

1. 待用户确认
2. 暂时保留的技术债
3. 已识别但未处理的风险
4. 下次继续时应该先检查什么

问题关闭后及时移除或标记已解决。

## 写法要求

1. 优先写事实、决策和约束，不写过程流水账
2. 优先写“为什么”和“以后谁会踩坑”，不是重复 diff
3. 每条记录都要让未来的 AI 能独立理解，不依赖当前对话
4. 内容保持短、硬、可检索
5. 同一信息只放一个最合适的文件，其他地方用一句话引用

## 推荐工作流

1. 开始任务时先读 `docs/ai-memory/README.md`
2. 实现前补齐必要上下文
3. 完成代码后判断是否触发更新条件
4. 更新对应 memory 文件
5. 在最终回复里简要说明本次同步了哪些 memory 文件

## 自检清单

提交前自查：

1. 这次改动是否留下了未来会丢失的重要信息
2. 如果三天后在压缩上下文里继续做，这条记录是否足够让我快速接手
3. 是否把短期细节误写进长期上下文
4. 是否把同一件事重复写进多个文件
